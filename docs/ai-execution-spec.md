# AI 执行规范

## 1. 执行目标

每一轮都必须形成一个“闭环能力单元”，至少包含：

- 一个新的确定性、可测试能力
- 一个明确标注的能力差距
- 代码、测试、文档三者同时落地

## 2. 覆盖要求

每一轮都必须包含：

- 能力测试：验证正常路径
- 边界测试：验证边界条件
- 差距测试：只要存在歧义或未支持场景，就必须保留

所有行为都必须满足：

- 确定性
- 可复现
- 跨运行稳定

## 3. 宽度要求

每一轮都必须：

- 完成一个 P1 目标
- 并且至少触及一个 P2 / P3 目标

禁止只围绕单一模块做局部收尾。

## 4. 能力与差距纪律

- 能力：稳定、确定、已被测试覆盖的行为
- 差距：明确未支持或语义未闭合的行为

禁止：

- 把不稳定逻辑伪装成能力
- 没有诊断信息的静默回退

## 5. 必须更新的文档

每一轮都必须同步更新：

- `docs/session-handoff.md`
- `docs/next-task-prompt.md`
- `docs/test-capability-coverage.md`
- `docs/design-doc-sync-tracker.md`

## 6. 代码约束

- 不修改已发布 API contract
- 不引入 breaking change
- 允许重构内部 helper

## 7. 诊断要求

如果模块支持诊断：

- 歧义发生时必须给出解释或问题描述
- 结果与诊断必须一致

## 8. 禁止行为

- 只写代码不写测试
- 只实现 happy path
- 没有标记差距就做部分实现
- 声称已支持但没有覆盖
- 产生静默且非确定性的行为
