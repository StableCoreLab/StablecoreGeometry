# Geometry SDK 与二进制发布设计文档

## 1. 文档目的

本文档用于定义几何库未来作为 `DLL + import lib + headers` 发布时的边界规则。

本文档重点解决以下问题：

- 模板类型是否直接作为 DLL 稳定接口
- 哪些类型属于库内部泛型实现层
- 哪些类型属于对外 SDK 导出层
- 是否允许跨 DLL 边界暴露 STL 容器、虚类和模板类
- 如何在不推翻当前设计的前提下，为后续发布形态留出正确演进路径

本文档是 `library-design.md` 在“工程发布形态”和“ABI 边界”层的细化补充。

## 1.1 文档关系

本文档与其它设计文档的关系如下：

- `library-design.md` 负责定义整个几何库的分层、阶段目标和总边界
- `point-vector-design.md`、`segment-design.md`、`box-design.md`、`polyline-design.md`、`polygon-design.md` 负责定义几何语义本身
- 本文档负责定义这些几何语义未来如何安全地暴露为 SDK 和 DLL 接口

执行要求：

- 几何语义文档可以保留模板化设计
- 但任何面向外部程序的稳定二进制接口，都必须受本文档约束
- 若未来要发布正式 SDK，则必须先满足本文档中的导出边界要求，而不是直接把当前头文件内部实现整体暴露出去

## 2. 核心结论

当前库内部允许保留模板类型体系，但未来 DLL 对外接口不应直接以模板体系作为稳定 SDK 边界。

执行结论：

- 库内部可以继续存在 `Point2<T>`、`Segment2<T>`、`Box2<T>`、`Polyline2<T>`、`Polygon2<T>`
- 但 SDK 导出层第一版应统一收敛到 `double` 主路径
- 对外发布时，优先导出 `Point2d`、`Vector2d`、`LineSegment2d`、`ArcSegment2d`、`Box2d`、`Polyline2d`、`Polygon2d`
- 不把“任意模板实例都可直接跨 DLL 使用”作为设计目标

这条结论必须尽早固定，否则后续头文件组织、实现组织和 ABI 约束都会持续漂移。

## 3. 为什么模板类型不适合作为 DLL 稳定接口

模板类型本身并不是不能和 DLL 共存，但它不适合作为稳定对外二进制接口的主体。

原因如下：

- 模板通常要求调用方在编译期看到完整实现
- 这会迫使大量实现继续放在头文件里
- 一旦调用方自己实例化模板，就很难把实现真正收进 DLL
- 模板实例集合如果不受控，会导致导出边界膨胀
- 不同编译器、编译选项、STL 实现下，跨模块 ABI 更脆弱

因此如果目标明确是：

- 生成 DLL
- 给别人 `headers + lib + dll`
- 让别人以 SDK 方式接入

那就不能把“模板头文件自然外露”当成最终发布形态。

## 4. 当前推荐架构

未来发布形态统一分三层：

### 4.1 内部几何语义层

这一层允许保留当前模板设计：

- `Point2<T>`
- `Vector2<T>`
- `Segment2<T>`
- `LineSegment2<T>`
- `ArcSegment2<T>`
- `Box2<T>`
- `Polyline2<T>`
- `Polygon2<T>`

这一层的职责是：

- 验证几何抽象是否合理
- 支撑内部算法实现
- 保持类型系统表达力

这一层不是对外稳定 ABI 承诺层。

### 4.2 内部实现收敛层

这一层负责把模板实现收敛到受控实例。

第一版执行建议：

- 只把 `double` 作为正式算法主路径
- `int` 等其它模板实例只作为内部表达、测试或局部辅助
- 若某个类型确实需要编译进库体，则显式实例化 `double` 版本

这层的核心目标是：

- 防止模板实例无边界扩散
- 为 DLL 导出层准备单一精度主路径

### 4.3 SDK 导出层

这一层才是未来给外部程序的稳定接口层。

第一版执行结论：

- SDK 统一导出 `double` 专用几何类型
- SDK 不承诺任意模板参数版本
- SDK 头文件不要求调用方重新编译大量模板实现
- 简单值类型直接导出实现类或结构体
- 复杂对象导出稳定壳类，内部使用 pImpl
- 算法能力优先以自由函数提供，而不是因为算法复杂就对象化
- 对简单值类型，优先直接以模板类的 `double` 实例作为 SDK 正式类型

建议导出类型：

- `Point2d`
- `Vector2d`
- `LineSegment2d`
- `ArcSegment2d`
- `Box2d`
- `Polyline2d`
- `Polygon2d`

其中应进一步区分：

- `Point2d`、`Vector2d`、`Box2d` 这类简单值类型，直接导出实现类，并优先由模板类的 `double` 实例承担该角色
- `Polyline2d`、`Polygon2d` 这类复杂对象，未来若进入 SDK，应使用 pImpl
- `LineSegment2d` 作为简单值类型可直接导出
- `ArcSegment2d` 也可直接导出，但公开表示统一收敛为 `center + radius + startAngle + sweepAngle`

`ArcSegment2d` 的 SDK 公开规则固定为：

- `center` 为圆心
- `radius > 0`
- `startAngle` 单位为弧度
- `sweepAngle` 单位为弧度
- `sweepAngle > 0` 表示逆时针
- `sweepAngle < 0` 表示顺时针
- 第一版要求 `0 < |sweepAngle| <= 2pi`
- `endAngle` 作为导出语义上的派生量定义为 `startAngle + sweepAngle`

## 5. 第一版 SDK 精度策略

第一版 SDK 统一采用 `double` 主路径。

执行结论：

- `double` 是对外正式发布精度
- `int` 不是 DLL 稳定接口承诺的一部分
- 若未来需要整数输入场景，应通过构造、转换或单独 API 支持，而不是直接把所有 `T` 暴露到 SDK 边界

这样做的原因是：

- 几何算法主路径本就更适合浮点计算
- ABI 更清晰
- 文档、测试和导出策略都更容易收敛
- 更符合未来算量产品主场景

## 6. 跨 DLL 边界的禁止项

为了保证 SDK 稳定性，跨 DLL 边界应尽量避免直接暴露以下内容：

- 任意模板类
- 未收敛实例的模板函数
- STL 容器作为长期稳定 ABI 承诺
- 依赖编译器布局细节的复杂模板嵌套类型
- 以内部虚继承层直接作为公开 ABI 主体

执行建议：

- 当前内部可以继续用 `std::vector`、`std::shared_ptr`、抽象基类和模板
- 但 SDK 层不要直接把这些内部组织方式当成对外稳定承诺

这意味着：

- 当前 `Polyline2<T>` 内部采用拥有式异构段容器是允许的
- 但未来 SDK 是否直接暴露该内部容器，不应现在就锁死

进一步执行约束：

- 不能仅因为某个算法实现复杂，就把本来应为值类型的对象改成 pImpl
- 是否采用 pImpl，应看对象的内部表示、生命周期、可变性和 ABI 演进风险

## 7. 对当前代码设计的影响

当前这套文档和代码设计不需要立刻推翻，但必须增加一个新的工程边界：

- “模板设计”不等于“未来 SDK 直接发布形态”

对现阶段的具体影响如下。

### 7.1 类型设计可以继续模板化

当前 `types` 层可以继续保持模板形式，用于把几何语义先设计和实现清楚。

### 7.2 算法实现应优先围绕 `double`

即使保留模板类型，算法主路径也应优先保证：

- `Point2d`
- `Segment2d`
- `Box2d`
- `Polyline2d`
- `Polygon2d`

这会直接降低未来 DLL 收敛成本。

### 7.3 SDK 类型分层规则

后续 SDK 类型设计统一按以下方式收敛：

- 第一类：简单值类型
  - 例如 `Point2d`、`Vector2d`、`Box2d`
  - 直接导出实现类或结构体
  - 优先直接复用模板类的 `double` 实例
  - 不再额外声明一套与模板实例并行的导出专用类型
  - 允许提供少量内联方法
- 第二类：复杂几何对象
  - 例如 `Polyline2d`、`Polygon2d`
  - 导出稳定接口类
  - 内部表示通过 pImpl 隐藏
- 第三类：算法能力
  - 例如距离、投影、包含、相交判断
  - 优先导出自由函数
  - 只有在必须绑定复杂上下文、缓存或状态时才考虑算法对象化

### 7.4 文档不再把“所有模板实例都属于正式发布目标”作为默认前提

后续文档如果提到类型别名，应明确区分：

- 内部模板语义
- 对外 SDK 导出形态

对于简单值类型，执行收敛方式应是：

- `Point2d`、`Vector2d`、`Box2d` 直接成为 SDK 正式类型名
- 这些类型优先由 `Point2<double>`、`Vector2<double>`、`Box2<double>` 承担
- 不再长期维护额外 `Geo*` 命名层

## 8. 当前不建议立即做的事情

当前阶段不建议立即为了 DLL 发布而做以下大改：

- 立刻删除全部模板设计
- 立刻把所有类型改成纯 C API
- 立刻把所有内部容器和抽象层重写成 ABI 专用结构

原因是：

- 你现在仍处于几何语义和算法边界快速收敛阶段
- 过早为 ABI 做到底，会显著拖慢主线设计

当前更合理的做法是：

- 先把几何语义层做稳
- 同时从现在开始约束未来导出边界

## 9. 后续演进建议

未来从内部库走向正式 SDK，建议按以下顺序推进：

1. 保持内部模板层，但收敛算法主路径到 `double`
2. 为核心类型补一层受控导出接口
3. 把 SDK 面拆成“值类型 / 复杂对象 / 算法函数”三类
4. 明确哪些复杂对象使用 pImpl，哪些简单类型直接导出
5. 明确哪些 API 属于 SDK，哪些仅属于内部实现
6. 在需要时再设计更稳定的 ABI 层，例如：
   - 非模板导出类
   - pImpl
   - handle API
   - C ABI 包装层

第一版不要求一步到位，但必须沿这个方向收敛，不能继续把模板头文件自然外露当成最终交付模式。

## 10. 序列化、调试与测试约定

为了支持未来图元数据保存、问题排查和产品自动化测试，SDK 设计还应额外遵守以下工程约定。

### 10.1 序列化放置规则

不要求所有几何类都内建成员形式的序列化与反序列化方法。

执行结论：

- 序列化与反序列化优先放在独立 `serialize/` 或 `io/` 模块
- 核心几何类型优先保持几何语义本身，不直接承担持久化格式职责
- 序列化的对象是稳定语义数据，不是运行时缓存、索引、加速结构或 `pImpl` 内部状态

简单值类型采用轻量规则：

- `Point2d`、`Vector2d`、`Box2d` 允许采用“外置但近旁”的轻量序列化组织
- 即类型本身不内嵌格式逻辑，但对应序列化代码应尽量靠近类型定义，降低维护摩擦
- 这类对象仍不建议长期维护成员形式 `ToJson()`、`FromJson()` 作为主路径

复杂对象采用严格外置规则：

- `Polyline2d`、`Polygon2d` 以及未来复杂对象的序列化必须放在独立序列化模块
- 不把格式版本、兼容逻辑、部分导入导出策略塞进对象本体或 `pImpl` 接口

为降低“改类型忘改序列化”的风险，应同时执行：

- 每个可序列化类型只保留单一正式序列化入口
- 新增或调整持久化字段时，必须同步更新对应序列化代码
- 每个可序列化类型都必须具备 round-trip 测试

### 10.2 调试最小接口

为了后续日志排查、问题复现和人工检查，对外几何类型应提供最小调试能力。

所有稳定对外类型至少应支持：

- `IsValid()`
- 稳定、可读的调试输出能力，例如 `DebugString()` 或外部 `ToDebugString(...)`

简单值类型建议再提供：

- 精确比较能力，例如 `operator==` 或 `Equals(...)`
- 浮点近似比较能力，例如 `AlmostEquals(...)`

复杂对象建议再提供：

- 基本结构观测能力，例如 `PointCount()`、`SegmentCount()`
- 几何摘要能力，例如 `Bounds()`
- 必要时提供类型识别能力，例如 `Kind()` 或等价机制

执行约束：

- 调试输出应稳定、确定，适合作为日志和测试失败信息
- 调试接口只暴露对定位问题有价值的信息，不泄露 `pImpl` 内部实现细节

### 10.3 自动化测试预留接口

为了支持单元测试、回归测试、产品级批量验证和未来回放机制，稳定对外类型和算法入口应预留以下标准能力：

- `IsValid()`，用于对象状态和前置条件检查
- `AlmostEquals(...)` 或等价近似比较机制，处理浮点几何比较
- 稳定调试输出，用于失败信息和快照对比
- 规范化的序列化能力，用于样例固化和回放

测试支持层应逐步补齐：

- 几何对象近似断言，例如点、向量、包围盒、多段线、多边形比较
- round-trip 测试支持，即 `deserialize(serialize(x))` 后与原对象等价
- 非法输入和缺字段反序列化测试
- 常见图元样例构造器
- 属性测试或批量回归测试入口

执行原则：

- 测试依赖稳定语义，不依赖对象内部实现布局
- 快照和回放优先使用统一序列化格式，不直接依赖临时日志文本
- `pImpl` 对象的测试以公开接口、公开序列化结果和公开调试能力为准，不检查内部表示

## 11. 当前执行结论

后续开发必须按以下原则执行：

- 内部可以保留模板化几何类型体系
- DLL 对外发布层不以模板体系作为稳定 ABI
- 第一版 SDK 导出统一收敛到 `double` 主路径
- 当前代码实现允许继续服务模板设计，但后续算法和接口优先围绕 `double` 收敛
- 简单值类型直接导出实现类
- 简单值类型优先直接收敛到模板类的 `double` 实例，不再额外维护导出专用类型
- 复杂对象使用稳定壳类 + pImpl
- 算法优先保持为自由函数，而不是因为算法复杂而类化
- 未来若进入正式 SDK 阶段，必须单独设计导出层，而不是直接把当前内部头文件整体外发

本文档即为后续几何库发布形态与二进制接口设计依据。

## 12. 当前落地入口

当前仓库已经补齐一个最小的 SDK 入口层，建议外部调用方优先包含：

```cpp
#include "sdk/Geometry.h"
```

当前实际落地的第一版 SDK 面是：

- `geometry::sdk::Point2d`
- `geometry::sdk::Vector2d`
- `geometry::sdk::Box2d`
- `geometry::sdk::SegmentProjection2d`
- `geometry::sdk::Polyline2d`
- `geometry::sdk::Polygon2d`
- `Distance`、`DistanceSquared`
- `ProjectPointToSegment`
- `Contains`、`Intersects` 的 box 查询入口

当前已为 `Polyline2d`、`Polygon2d` 预留 SDK 壳接口，但不把内部多态段容器作为对外 ABI 承诺。后续实现仍应采用 pImpl，而不是直接暴露当前内部结构。
